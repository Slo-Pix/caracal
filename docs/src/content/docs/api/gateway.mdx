The Gateway is the network entry point for protected upstream calls, including MCP tool calls. It does not expose a resource API — every inbound path is proxied to a configured upstream. The Gateway runs on port `8081` (fixed). TLS is configured with `TLS_CERT_FILE` and `TLS_KEY_FILE`; plaintext HTTP is allowed only for internal development or internal service-network deployments.

---

## Inbound request authentication

The Gateway applies five checks in order. The first failure halts the request.

### 1. Bearer token extraction

Reads `Authorization: Bearer <token>`. Returns `401` if the header is missing, not in Bearer format, or exceeds 4 096 bytes.

### 2. JWT expiry preflight

Peeks the `exp` claim without signature verification. Rejects the token with `401 CredentialExpired` if it would expire within 35 seconds. This prevents issuing an upstream token that arrives stale.

### 3. Zone/resource binding lookup

Reads the token's `zone_id` claim and the `X-Caracal-Resource` header, then looks up the Gateway binding by `(zone_id, resource_identifier)`. The unverified `zone_id` only selects the binding and JWKS namespace; trust is established by the following signature check. If no binding exists, the Gateway returns `403 AccessDenied`.

### 4. Signature verification

Fetches the zone's JWKS from `{STS_URL}/.well-known/jwks.json?zone_id=<zone>`. Verifies the ES256 signature using the key matching the `kid` header. JWKS responses are cached 5 minutes per zone. Returns `401 InvalidToken` on failure.

### 5. JTI replay detection

Checks whether the token's `jti` claim has been seen. Tokens with `"use": "ambient"` are reusable; tokens with `"use": "per_call"` are single-use. Single-use JTIs are stored in Redis with a TTL equal to the token's remaining lifetime. Returns `401 InvalidToken` if a per-call token is replayed.

### 6. Authority revocation check

Extracts the `sid`, `agent_session_id`, and `delegation_edge_id` claims and checks the in-memory revocation store. Returns `401 InvalidToken` if the STS session, agent session, or delegation edge is revoked.

## Required inbound headers

| Header | Required | Description |
| --- | --- | --- |
| `Authorization` | Yes | `Bearer <ambient-token>` |
| `X-Caracal-Resource` | Yes | Resource identifier used to look up the upstream binding |

**Forbidden:** Clients must not send `X-Caracal-Client-ID`. The Gateway strips it and returns `400 InvalidToken` if present.

The `X-Request-Id` header is preserved if valid (alphanumeric, dots, hyphens, colons, 1–128 characters) or replaced with a generated UUIDv7.

## Token exchange with the STS

After authentication, the Gateway exchanges the caller's ambient token for a per-resource mandate by calling the STS.

**Request:**

```
POST {STS_URL}/oauth/2/token
Content-Type: application/x-www-form-urlencoded

grant_type=urn:ietf:params:oauth:grant-type:token-exchange
subject_token=<ambient-bearer>
subject_token_type=urn:ietf:params:oauth:token-type:access_token
zone_id=<verified-zone>
application_id=<app-from-binding>
resource=<resource-identifier>
```

The Gateway uses the zone/resource binding stored in its database for the resource identified by `X-Caracal-Resource`. The STS exchange is authenticated with the Gateway↔STS HMAC key, so only the Gateway can receive provider-native upstream directives.

**Timeout:** `STS_TIMEOUT` (default `5s`). Returns `504 Gateway Timeout` if exceeded.

## Headers forwarded to the upstream

After a successful exchange, the Gateway sends the request to the upstream URL returned by the STS.

**Authorization header:**

| `auth_mode` | Header sent |
| --- | --- |
| `caracal_jwt` | `Authorization: Bearer <sts-issued-mandate>` |
| `provider_oauth` | `{auth_header}: {auth_scheme} {provider_token}` |
| `provider_apikey` | `{auth_header}: {provider_token}`, or `{auth_scheme} {provider_token}` when `auth_scheme` is configured |

When `auth_mode` is `provider_oauth` or `provider_apikey`, the inbound `Authorization` header is removed before forwarding. The Caracal mandate is not forwarded to provider-native upstreams by default. `X-Caracal-Identity: <mandate>` is sent only when the STS upstream directive sets `forward_caracal_identity: true`, which should be reserved for trusted internal services, controlled MCP servers, or sidecars that intentionally verify Caracal identity.

Provider-native credentials are never returned to normal STS clients. They appear only in the internal Gateway-authenticated exchange response and are immediately substituted into the upstream request.

**Additional forwarded headers:**

| Header | Value |
| --- | --- |
| `X-Request-Id` | Preserved or generated request ID |
| `Traceparent` | W3C trace context derived from the request ID |
| `X-Forwarded-For` | Client IP (replaced, never appended) |
| `X-Forwarded-Proto` | `https` or `http` |
| `X-Forwarded-Host` | Original `Host` header |
| `X-Caracal-Identity` | Optional; forwarded only when `forward_caracal_identity` is true |
| `X-Caracal-Token-Expires-In` | Seconds until the mandate expires (set on the response) |

**Stripped or replaced before forwarding:** inbound `Authorization`, `X-Caracal-Client-ID`, `X-Caracal-Resource`, `X-Caracal-Upstream`, `X-Caracal-Identity`, the selected upstream auth header, and all RFC 7230 hop-by-hop headers.

The upstream URL is constructed by joining the upstream base path with the inbound request path. Query strings are merged; upstream values take precedence on conflicts. Path traversal attempts (`../`) are rejected with `400 InvalidToken`.

---

## Upstream SSRF guard

Before forwarding, the Gateway validates the upstream URL returned by the STS:

- The host must not resolve to a loopback or private IP address (unless `ALLOW_PRIVATE_UPSTREAMS=true`).
- The host must appear in `UPSTREAM_HOST_ALLOWLIST` if that variable is set.

Violations return `502 Internal` with a sanitized client message.

---

## Action audit

For each authorized upstream attempt, the Gateway emits a `gateway_resource_request` audit event to `caracal.audit.events`. The event includes the request ID, zone, resource identifier, Gateway application ID, upstream host, auth mode, HTTP method, gateway/upstream status, latency, and a short subject-token fingerprint.

Audit metadata deliberately excludes authorization headers, provider tokens, request bodies, response bodies, full URLs, and query strings. If Redis is temporarily unavailable, events are buffered and persisted under `AUDIT_REPLAY_DIR` for replay on restart.

## Response handling and revocation-aware streaming

The Gateway streams the upstream response body to the caller. Every 4 KB of streamed body, it checks the session revocation store. If the session is revoked mid-stream:

- The response body is truncated.
- The HTTP trailer `X-Caracal-Revoked: true` is set to signal the truncation.

This mechanism is a best-effort guard — clients should check for the trailer and retry if they receive a truncated response.

---

## Revocation store

The Gateway maintains an in-memory revocation cache populated from the `caracal.sessions.revoke` Redis stream. Stream consumer details:

| Parameter | Value |
| --- | --- |
| Stream | `caracal.sessions.revoke` |
| Consumer group | `gateway-revocation` |
| Consumer ID | `gateway-{hostname}-{pid}` |
| Batch size | 50 messages |
| Block duration | 1 second |
| Pending reclaim | `XAUTOCLAIM` with 30 s idle window |

Dead-letter messages (invalid HMAC signatures) are moved to `caracal.sessions.revoke.dead`.

In-memory entries are pruned every 30 minutes. Revocation entries have a 24-hour TTL (by which time any mandate referencing the session has expired).
