The Gateway is the Caracal service intended to sit on the protected resource boundary. It verifies inbound mandates, exchanges a fresh per-call mandate, blocks unsafe upstream targets, and can serve HTTPS directly when certificate files are configured.

Use `CARACAL_MODE=stable` for stable deployments, `CARACAL_MODE=rc` for rc deployments, and `CARACAL_MODE=dev` for development. rc and stable enable safety checks in service code; they do not replace network-level hardening, TLS termination, secret management, or private service networking.

---

## TLS configuration

The Gateway serves HTTPS when both certificate files are set:

```bash
TLS_CERT_FILE=/etc/caracal/tls/server.crt
TLS_KEY_FILE=/etc/caracal/tls/server.key
```

Both variables must be set together. Setting only one causes startup to fail:

```text
TLS_CERT_FILE and TLS_KEY_FILE must both be set
```

When neither variable is set, the Gateway listens over plain HTTP. That is acceptable for the local Docker Compose stack and for deployments where a trusted load balancer or ingress terminates TLS before traffic reaches the Gateway. If the Gateway is exposed directly, configure `TLS_CERT_FILE` and `TLS_KEY_FILE`.

**Minimum TLS version:** TLS 1.2, enforced by `tls.Config{MinVersion: tls.VersionTLS12}` when Gateway TLS is enabled. Cipher suite selection uses Go's defaults for TLS 1.2 and TLS 1.3.

**Certificate format:** Standard PEM. Include the leaf certificate and any required intermediates in `TLS_CERT_FILE`.

## Runtime mode

`CARACAL_MODE` accepts exactly two values:

| Value | Use |
| --- | --- |
| `dev` | Local repository and Docker Compose development |
| `runtime` | Packaged or production-like runtime deployments |

If unset, Go service configuration defaults to `runtime`; the local compose stack sets `CARACAL_MODE=dev`.

Runtime mode enforces these Gateway checks at startup:

| Check | Runtime behavior |
| --- | --- |
| `INSECURE_HTTP` | Must not be truthy |
| `INSECURE_STS` | Must not be truthy |
| `JTI_FAIL_OPEN` | Must not be `true` |
| `PORT` | Must be `8081` |
| `REDIS_URL` | Required |
| `STREAMS_HMAC_KEY` | Required |
| `AUDIT_HMAC_KEY` | Required |
| `UPSTREAM_HOST_ALLOWLIST` | Required when `ALLOW_PRIVATE_UPSTREAMS=true` |
| `STS_URL` | Must be absolute; HTTP is allowed only for internal hosts such as `sts`, `localhost`, or loopback |

Any value other than `dev` or `runtime` is rejected.

## Development escape hatches

Two flags exist for local development compatibility:

| Variable | Effect | Forbidden when |
| --- | --- | --- |
| `INSECURE_HTTP=true` | Marks plaintext Gateway operation as intentional | `CARACAL_MODE=rc` or `CARACAL_MODE=stable` |
| `INSECURE_STS=true` | Marks plaintext STS calls as intentional | `CARACAL_MODE=rc` or `CARACAL_MODE=stable` |

Do not set either flag in deployments that handle real credentials or tokens.

## SSRF defenses

The Gateway proxies requests to upstream resource URLs loaded from Postgres. To prevent server-side request forgery:

**Private upstream blocking (default):**

`ALLOW_PRIVATE_UPSTREAMS=false` prevents forwarding to private IP ranges and loopback addresses. Requests to blocked upstreams fail before proxying.

**Private upstream allowlist:**

If the Gateway must reach internal upstream services, set:

```bash
ALLOW_PRIVATE_UPSTREAMS=true
UPSTREAM_HOST_ALLOWLIST=api.internal.company.com,payments.internal.company.com
```

When `CARACAL_MODE=rc` or `CARACAL_MODE=stable` and private upstreams are allowed, `UPSTREAM_HOST_ALLOWLIST` is required. Requests to hosts outside the allowlist are rejected.

**Path traversal protection:**

Gateway upstream URL validation rejects paths containing traversal sequences before proxying.

## JTI replay protection

The Gateway tracks JWT IDs (`jti` claims) in Redis to prevent replay of intercepted mandates. Redis is required at startup.

| Setting | Behavior |
| --- | --- |
| `JTI_FAIL_OPEN=false` | Reject requests if Redis JTI lookup fails; default |
| `JTI_FAIL_OPEN=true` | Allow requests through when Redis JTI lookup fails; forbidden when `CARACAL_MODE=rc` or `CARACAL_MODE=stable` |

## Request size and timeout controls

| Variable | Default | Description |
| --- | --- | --- |
| `MAX_REQUEST_BYTES` | `10485760` | Maximum request body size; larger requests are rejected with 413 |
| `STS_TIMEOUT` | `5s` | Timeout for STS token exchange calls |
| `UPSTREAM_TIMEOUT` | `30s` | Timeout for proxied upstream requests |
| `READ_HEADER_TIMEOUT` | `5s` | Time allowed to read request headers |
| `READ_TIMEOUT` | `30s` | Time allowed to read the full request |
| `WRITE_TIMEOUT` | `60s` | Time allowed to write the response |
| `IDLE_TIMEOUT` | `120s` | Keep-alive idle timeout |

Tune `UPSTREAM_TIMEOUT` to the protected service's P99 response time. A timeout below expected upstream latency causes false 504 responses.

## Hardening checklist

- Set `CARACAL_MODE=stable`.
- Provide `TLS_CERT_FILE` and `TLS_KEY_FILE`, or terminate TLS at a trusted ingress before traffic reaches the Gateway.
- Keep `INSECURE_HTTP`, `INSECURE_STS`, and `JTI_FAIL_OPEN` unset or explicitly `false`.
- Set `STREAMS_HMAC_KEY` and `AUDIT_HMAC_KEY` to 32-byte hex secrets.
- Keep `ALLOW_PRIVATE_UPSTREAMS=false` unless the Gateway must reach private upstreams.
- Set `UPSTREAM_HOST_ALLOWLIST` when private upstreams are enabled.
- Keep API, STS, Coordinator, Audit, Postgres, and Redis on private networks.
- Expose only the Gateway or its fronting ingress to external callers.
- Rotate TLS certificates before expiry and rotate shared secrets through your secret manager.

---

## Inter-service communication

In the Docker Compose stack, services communicate over Docker's internal network using container names such as `http://sts:8080` and `http://api:3000`. The Gateway connects directly to STS, Postgres, and Redis; inter-service traffic does not pass through the Gateway.

In production orchestration, keep service-to-service traffic on private networks and enforce network policy or mTLS at the platform layer where required.
