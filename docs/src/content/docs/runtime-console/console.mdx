The web console is the supported browser UI for setting up and operating protected agent access. The packaged `web` service serves the built SPA and a session-guarded backend-for-frontend from the same origin, then proxies console API calls to the internal API and Coordinator. For local Compose, open [http://localhost:3001](http://localhost:3001) after `caracal up`; use `caracal web` when you need the development launcher.

## Local Access

The web console expects the local stack to be running:

```bash
caracal up
caracal status --ready
```

Open [http://localhost:3001](http://localhost:3001). Compose maps `127.0.0.1:3001` to the BFF container on port `3002`.

Use the development launcher only when working on the console or auth BFF locally:

```bash
caracal web
```

The launcher serves on the same port as the packaged console ([http://localhost:3001](http://localhost:3001)), with the backend-for-frontend on `3002`. While it runs it stops the packaged web container so the two never contend for the port. Stopping the launcher (Ctrl+C) tears down both the UI and the backend; the packaged web container stays stopped, so run `caracal up` to bring the packaged console back.

Use `--allow-offline` with the development launcher when you only need sign-in or UI-only checks and the stack is not ready:

```bash
caracal web --allow-offline
```

## Development Launcher Options

| Option | Purpose |
| --- | --- |
| `--web-port <port>` | Bind the browser UI to a specific port (default `3001`). |
| `--auth-port <port>` | Bind the session/auth backend to a specific port (default `3002`). |
| `--build` | Build the web assets before launching. |
| `--allow-offline` | Start the UI even when local Caracal services are not ready. |

## Credentials and Endpoints

| Input | Purpose |
| --- | --- |
| `CARACAL_API_URL` | Cloud/custom admin API URL override. |
| `CARACAL_ADMIN_TOKEN` or `CARACAL_ADMIN_TOKEN_FILE` | Admin bearer token for management views. |
| `CARACAL_COORDINATOR_URL` | Cloud/custom Coordinator URL override for agent and delegation views. |
| `CARACAL_COORDINATOR_TOKEN` or `CARACAL_COORDINATOR_TOKEN_FILE` | Coordinator token for agent and delegation views. |

For BFF deployment variables such as `CARACAL_AUTH_URL`, `CARACAL_AUTH_DATABASE_URL`, `CARACAL_AUTH_SECRET`, `CARACAL_WEB_ROOT`, and `CARACAL_AUTH_AUTO_MIGRATE`, use [Configure Service Environment](/operations/env-vars/#web-console-bff).

For default local services, generated secret files from `caracal up` are preferred over exported token values so local reset cycles do not keep stale shell credentials. Those files live outside the source workspace by default and must not be mounted into agent workspaces.

## Browser Model

Open the packaged or development URL in a browser, sign in to the session-guarded UI, and navigate by sections instead of terminal keys.

| Section | What it manages |
| --- | --- |
| **Guided setup** | First zone, agent app, provider, resource, access policy, and runtime profile. |
| **Caracal Operator** | Operate the control plane from natural-language intent: describe a change, review the resolved plan, and apply it within your operator scope. |
| **Zones** | Zone records and active zone selection. |
| **Applications** | Confidential agent applications and one-time client secrets. |
| **Providers** | Gateway upstream auth modes for Caracal mandates, OAuth 2.0 authorization code, OAuth 2.0 client credentials, API-key, and bearer-token upstreams. |
| **Resources** | Protected resource identifiers, scopes, upstream URLs, Gateway application bindings, and upstream credential provider bindings. |
| **Policies** | Policy content and versions. |
| **Policy Sets** | Policy-set composition, simulation, activation, and shadow evaluation. |
| **Sessions** | Active authority-session records. |
| **Agents** | Agent sessions, lifecycle, lease health, and session trees. |
| **Delegation** | Authority edges between agent sessions. |
| **Audit** | Recent decisions, filters, event details, and request trace entry points. |
| **Request Traces** | Direct lookup for one known request ID. |
| **Control** | Control API exposure and Control credentials. |
| **Diagnostics** | Health, readiness, zone, and preflight checks. |

## Sessions Views

The session, agent, and delegation pages observe the runtime identity objects. Authority is minted by the STS, and delegation is authored through the agent SDK; the web console lists and inspects them.

| View | What it shows | Object |
| --- | --- | --- |
| **Sessions** | STS-minted subject sessions for a user or application | a `sessions` row |
| **Agents** | A spawned agent's runtime session, lifecycle, and lease health | an `agent_sessions` row |
| **Delegation** | Authority edges between agent sessions | a `delegation_edges` row |

An **authority session** is the root credential a principal holds after a token exchange. An **agent session** is one spawned agent under an application; a *delegated* session is an agent session with an inbound delegation edge. A **delegation** edge connects two agent sessions, so inbound and outbound edges are anchored on the agent session id.

The **Agents** page supports filtering by `status` (active, suspended, terminated), `lifecycle` (task, service), `application`, and `label`. The lease fields surface task TTL deadlines and service heartbeat deadlines. Use the session detail pages to view inbound and outbound delegation edges.

## Caracal Operator

Caracal Operator is a Community Edition console capability that lets you operate the control plane by describing what you want instead of navigating each page by hand. It turns intent into a reviewed plan, then applies it through the same guarded APIs you use manually — within your operator scope and recorded in the audit log. Every change it makes is one you could have made yourself; it introduces no new authority.

The Operator works against a capability catalog: the set of control-plane actions it can take, grouped by the objects you operate — zones, applications, providers, resources, access, and policy. Each capability is classified as read-only or state-changing, so a request that only inspects state is always distinguishable from one that changes it.

The Operator is a workspace, not a chat box. Each session shows an execution timeline of everything that has happened — your messages, the plans it proposed, your approvals, and the result of each applied step. A proposed plan appears as a card with its steps and effects; you approve, reject, or apply it inline, and the card updates to show progress as each step runs.

Work happens in a session — an auditable conversation scoped to the active zone. Within a session the Operator follows a governed lifecycle:

1. **Propose.** Intent becomes a plan whose every step is validated against the capability catalog. A step that names an unknown action or invalid arguments is rejected before anything runs.
2. **Preview.** The plan is resolved against your live state as a read-only dry run, so each step is marked as a create, an update, a no-op against an object that already exists, or blocked when a referenced object is missing. Nothing is written.
3. **Decide.** You approve or reject the plan. A plan is decided once, and only an approved plan is eligible to apply.
4. **Apply.** An approved plan is executed step by step. Before any write, the plan is re-validated and re-previewed against current state so drift since approval cannot apply a stale change. Each step is then applied as its own authenticated, audited control-plane call under the Operator's least-privilege identity. If a step fails, the plan stops there rather than pressing on, so it never blindly pushes past an error; the steps that already applied remain applied, and the apply response records exactly which steps succeeded and which one stopped the plan. So that a change which may already have taken effect can never be applied twice, a plan that has begun applying is not re-run, while a plan that fails before anything is applied stays available to retry. Each applied step is written to the conversation as an execution record, and one-time material such as a newly issued client secret is returned to you once and never stored.

A plan applies only once. Any one-time secret it produces is surfaced in the apply response and is never written to the conversation or the audit log.

### Operator authority and isolation

The Operator runs under its own reserved identity, distinct from the human operator who approves a plan. When a plan is applied, your approval is recorded in the admin audit log, while the execution record names the Operator's reserved principal as the authority that actually made the change — the Operator's authority is a delegated, attenuated form of your own.

That delegated authority is least-privilege: the Operator may execute only the capabilities explicitly granted to it, and the grant defaults to the small set of state-changing capabilities it is built to apply. A plan that asks the Operator to execute a capability outside its grant is refused as forbidden, before execution and regardless of whether a handler exists. The grant governs state-changing capabilities only; read-only capabilities change nothing and are always available.

The Operator is also bounded by a zone isolation boundary. An administrator can mark zones as system zones; the Operator will not open a session in, or execute against, a system zone at all, keeping its authority away from the infrastructure that runs Caracal itself.

Because the Operator acts within a delegated subset of your operator authority and never outside the active zone, the same row-level zone isolation and audit trail that govern manual changes also govern Operator changes. Nothing is applied until you approve the plan, every applied change is attributed to you, and the executing authority is the Operator's own least-privilege identity.

### Natural-language provider

Turning your words into a plan needs a language model, and that model is something you supply. The Operator is built on the [Vercel AI SDK](https://ai-sdk.dev) and talks to any OpenAI-compatible endpoint, so you can bring a hosted provider key, point at a local model server, or list several providers that are tried in order for resilience. For production the recommended backend is a self-hosted [LiteLLM](https://docs.litellm.ai) proxy, which owns per-provider integration, bring-your-own-key budgets, and local-model routing behind one OpenAI-compatible endpoint — so Caracal never maintains provider SDKs itself. Keys are read from the API service environment and never stored in or returned by the console.

The provider layer is optional and off by default. With nothing configured, the Operator runs without natural-language assistance and consumes no AI resources; the rest of the console — including everything you would do by hand — is unaffected. When you do configure providers, the AI panel shows them in failover order, and a connectivity check confirms a real request reaches your model before you rely on it.

Configure providers through the API service's `API_OPERATOR_AI_*` settings: list the provider ids in failover order, then give each one a base URL, a model, and, for hosted providers, a key.

### From words to a governed plan

When you send a message, the Operator first judges how much handling it needs and takes the lightest path that fits: a greeting or a question about what it can do gets a direct answer, a question about your current state or a past decision gets an explanation, and a request to change something becomes a plan — so a simple message is never forced through the planning pipeline. For a change, the language model proposes a plan, but it holds no authority — its proposal is only a draft. Caracal then validates that draft against the capability catalog, normalizes it to the catalog's own definitions, and previews it against your live state, exactly as it would a plan you wrote by hand. A proposal that names an unknown action, supplies bad arguments, or cannot be realized is recorded as a failed attempt, never as something you can approve.

The result is that natural language only ever produces a draft that enters the same propose → preview → decide → apply lifecycle. The model accelerates describing what you want; every safety property — validation, least-privilege authority, approval, audit — is enforced by Caracal, not by the model.

A request that combines several changes, or needs to know your current state before it can be planned, is handled as a composed request. The Operator first reads your live state through its own least-privilege, audited identity, plans against what it found, and then runs an advisory security review over the proposed plan — checking for over-grant, a write scope where a read would do, or a change that reaches wider than intended. The review is surfaced with the plan as a short summary and any findings, so you weigh them before deciding. It is advisory only: it never blocks or alters the plan, and the plan still flows through the same validation, preview, and approval as any other. A single, self-contained change skips the composition and stays the direct path.

Explanations are read-only. When you ask about your current state, the Operator first reads that state through the same governed control plane it would use to make a change — listing your applications, providers, resources, and policies under its own least-privilege, audited identity — and grounds its answer in what it just read rather than in a guess. Each read is governed independently, so a read it is not permitted simply narrows the evidence instead of failing the answer, and only the names and counts it found inform the response, never the underlying records. A question that needs no live state, such as what the Operator can do, is answered without any read. The Operator never makes a change while explaining, and will tell you to ask for a change so it can be planned and approved.

A read is answered by the assistant best suited to it. A question about why something was denied or why a change failed is handled as a diagnosis — the Operator reads your current state and recent activity and explains the most likely cause and a concrete next step. A question about how to connect a provider, or what a resource and its scopes should look like, is handled as integration guidance — the Operator translates the real-world product or API you name into the connection kind, resource, and scopes that express it in Caracal. Both are read-only: they explain and guide, and any change they point to still goes through the same plan, preview, and approval as everything else.

### Memory across a long session

A session can run long, so the Operator does not re-read the whole transcript each time. It keeps the recent activity in full and compresses everything before it into a small set of facts: which earlier plans were decided and how they turned out, how many changes have been applied, and which operations you have already rejected. This keeps the Operator's context accurate and its cost bounded no matter how long the conversation grows, and it means the Operator remembers your decisions — it will not re-propose an operation you have turned down unless you ask for it again.

## Guided Setup

Guided setup creates or selects the first working objects:

1. Zone
2. Agent app
3. Provider
4. Resource
5. Access policy
6. Review and create

Important guided fields include `upstream URL`, `first request path`, `runtime profile`, `write profile files`, `profile path`, `secret file`, `resource identifier`, and `resource scopes`.

Use file writing when you want the web console to create the runtime profile and secret file locally. Leave it disabled when a platform secret manager or CI system will provide those values.

## Troubleshooting

| Symptom | Check |
| --- | --- |
| Browser cannot reach the packaged web console | Confirm `caracal up` is running and no other service is using host port `3001`. |
| Browser cannot reach the development launcher | Confirm the `caracal web` process is still running and no other service is using the selected ports. |
| Product views show connection errors | Run `caracal status --ready`; custom deployments should confirm the admin API URL override. |
| `401 Unauthorized` | Check `CARACAL_ADMIN_TOKEN` or `CARACAL_ADMIN_TOKEN_FILE`. |
| `403 Forbidden` | The token lacks the required management scope. |
| Coordinator-backed views fail | Confirm the Coordinator token and, for custom deployments, the Coordinator URL override. |
| A zone-scoped page has no data | Select a zone before opening zone-scoped views. |
