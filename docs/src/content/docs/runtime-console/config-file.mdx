`caracal run` and the TypeScript, Python, and Go SDKs load workload settings from platform environment variables or from the same generated runtime profile. Use Console guided setup to generate the first profile from real zone, application, resource, policy, and secret-file values. `caracal.toml` is a portable profile contract; it is not Console state and it does not create, select, or manage zones.

The runtime validates configuration before token exchange. Unknown fields are rejected, credential environment names must be safe process environment keys, and non-local `http://` STS endpoints are rejected outside development unless `CARACAL_ALLOW_INSECURE_CONFIG_URLS=true` is set explicitly.

## Production configuration

Production and cloud workloads should avoid inline secrets and provide config through the platform:

| Variable | Required | Meaning |
| --- | --- | --- |
| `CARACAL_STS_URL` or `CARACAL_ZONE_URL` | yes | STS URL used for OAuth token exchange. Use HTTPS outside local development. |
| `CARACAL_ZONE_ID` | yes | Zone identifier. |
| `CARACAL_APPLICATION_ID` | yes | Confidential application identifier. |
| `CARACAL_APP_CLIENT_SECRET_FILE` | yes | Path to a mounted secret file containing the client secret. |
| `CARACAL_RUN_CREDENTIALS_FILE` | no | Path to a JSON credential manifest. |
| `CARACAL_RUN_CREDENTIALS` | no | Inline JSON credential manifest for ephemeral automation. |
| `CARACAL_RUN_CONTINUE_ON_FAILURE` | no | `true` or `false`; `true` is accepted only in development unless `CARACAL_ALLOW_REQUIRED_CREDENTIAL_FAILURE=true`. |
| `CARACAL_MCP_GOVERNANCE_MODE` | no | Platform override for MCP governance: `block` or `log`. `log` is accepted outside development only with `CARACAL_ALLOW_MCP_GOVERNANCE_LOG=true`. |

The credential manifest may be an array of required credentials, or an object with required credentials, optional credentials, and runtime policy. Each credential entry must use the real resource ID registered in Console and the token environment variable generated for that resource.

Mount secret files read-only where possible. The runtime rejects secret files that are group- or world-writable.

## Optional `caracal.toml`

### Lookup order

1. `CARACAL_CONFIG`, when set and the file exists.
2. `$XDG_CONFIG_HOME/caracal/caracal.toml`.
3. `~/.config/caracal/caracal.toml`.

The runtime intentionally does not read `./caracal.toml` from the current working directory because local config may contain credential material. Console, Doctor, and stack lifecycle commands do not load this file; it is workload runtime input for `caracal run` and SDK credential loaders.

### Permissions

On POSIX systems, the default config path must be owner-only:

```bash
chmod 600 ~/.config/caracal/caracal.toml
```

Explicit `CARACAL_CONFIG` paths may be read-only secret mounts, but they must not be group- or world-writable.

### Schema

| Field | Required | Meaning |
| --- | --- | --- |
| `zone_url` | yes | STS zone URL used by `caracal run` for OAuth token exchange. Local default is `http://localhost:8080`; use HTTPS outside development. |
| `sts_url` | no | SDK-readable STS alias. The runtime can use it as a fallback when `zone_url` is absent. |
| `coordinator_url` | yes for SDKs | Coordinator endpoint for clients that use generated profiles. The runtime ignores this field. |
| `gateway_url` | no | Gateway endpoint for SDK clients that route provider calls through Gateway. The runtime ignores this field. |
| `zone_id` | yes | Zone created in the Console. |
| `application_id` | yes | Confidential application used by the runner. |
| `app_client_secret_file` | yes, unless `app_client_secret` is set | Path to a secret file containing the confidential application secret. Preferred outside local development. |
| `app_client_secret` | yes, unless `app_client_secret_file` is set | Inline confidential application secret. Local development only. |
| `continue_on_failure` | no | Continue when required credential acquisition fails. Defaults to `false`; `true` is local-development only unless explicitly allowed by environment. |
| `credentials[]` | no | Required resource tokens to inject into the subprocess. |
| `optional_credentials[]` | no | Optional resource tokens with per-entry failure behavior. |
| `mcp_governance.mode` | no | `block` or `log` when the subprocess command references MCP server binaries. `log` is local-development only unless explicitly allowed by environment. |

Credential entries use:

| Field | Required | Meaning |
| --- | --- | --- |
| `env` | yes for `caracal run` | Environment variable name that receives the access token. SDKs use the same entry as a resource audience even when `env` is omitted. |
| `resource` | yes | Resource identifier registered in Caracal. |
| `upstream_prefix` | no | SDK-readable absolute URL prefix for Gateway-routed provider calls. The runtime ignores this field. |
| `on_failure` | optional | Optional credentials only: `warn` or `error`. |

`caracal run` exchanges the listed credentials and injects the resulting access tokens into the configured `env` names. SDKs use the same `credentials[]` and `optional_credentials[]` entries as token-exchange audiences; when `upstream_prefix` is present, SDK transports can also route matching calls through Gateway.

### Generated profiles

The Console guided setup result shows a complete profile with real values. Enable **write files** when you want the Console to write the profile and one-time secret file locally with owner-only permissions. If you leave file writing disabled, copy the generated profile to the shown profile path, store the one-time client secret in the shown secret-file path, and set owner-only permissions on both files. The checked-in `caracal.toml.example` is a reference schema, not an onboarding profile.

## Diagnose profile and runtime config

Use this sequence when `caracal run` or an SDK cannot find credentials:

1. Check whether `CARACAL_CONFIG` is set. If it is set, that exact file is the only runtime profile considered; a missing path fails instead of falling back.
2. Confirm the profile or platform environment contains `zone_id`, `application_id`, `app_client_secret_file` or `app_client_secret`, an STS URL, and every required credential resource.
3. Confirm the secret-file path contains the secret value, not the path variable itself, and that the file is not group- or world-writable.
4. Confirm each credential `env` is a safe process environment key and each `resource` is the real resource ID from Console.
5. Run `caracal status --ready` before debugging token exchange; readiness failures mean the stack cannot serve STS, Gateway, or audit reliably.
6. For SDK Gateway routing, confirm `gateway_url` is present and each Gateway-routed credential has the intended `upstream_prefix`.

Runtime shell errors name the source being validated: `environment`, the explicit `CARACAL_CONFIG` path, or the default profile path. Fix that source first; Caracal does not merge values from the other surfaces after one source wins.

## CI and secret mounts

For CI, prefer environment variables and secret files sourced from the CI secret store or orchestration secret mount. Use the zone ID, application ID, secret-file path, and credential manifest from the Console-generated setup or from your platform provisioning system. Use `CARACAL_CONFIG` only when a tool or workflow specifically needs a TOML profile.
