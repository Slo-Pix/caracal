When the active policy returns `{"step_up_required": "mfa"}` (or `"human_approval"` or `"software_attestation"`) in its `diagnostics`, the STS creates a step-up challenge and returns an `interaction_required` error response to the token exchange call. The client must keep the returned challenge proof, wait for an external party to satisfy the challenge, then retry the same exchange with `challenge_id` and `challenge_response`.

## Prerequisites

- The caller can make direct `application/x-www-form-urlencoded` requests to the STS token endpoint.
- The active policy includes a step-up trigger rule (see [Author a Rego Policy](./author-policy/)).
- An external system is capable of satisfying challenges via the control-plane API.

## What the STS returns on step-up

When a policy denies with `step_up_required` in diagnostics, the STS responds with HTTP 401:

```json
{
  "error": "interaction_required",
  "error_description": "Step-up challenge required",
  "challenge_id": "chall-abc123",
  "challenge_type": "mfa",
  "challenge_secret": "<base64url 32-byte secret>",
  "challenge_expires_at": "2026-05-22T12:50:00Z",
  "requestId": "req-def456"
}
```

The `challenge_secret` is returned only in this response. Store it only long enough to complete the retry. The STS stores only a SHA-256 hash of the secret and requires the original value as `challenge_response` on the follow-up token exchange.

## Satisfy a challenge

The external system — MFA provider, approval portal, or attestation service — calls the control-plane satisfaction endpoint with the approving subject:

```bash
curl -X POST \
  "$CARACAL_API_URL/v1/zones/$CARACAL_ZONE_ID/step-up-challenges/$CHALLENGE_ID/satisfy" \
  -H "Authorization: Bearer $CARACAL_ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"approver_subject_id":"user_approver_123"}'
```

The endpoint returns 200 with `id`, `satisfied_at`, and `approver_subject_id`. It returns 400 for an invalid body and 409 when the challenge cannot be satisfied because it is expired, already satisfied, or the approver is the same subject that triggered the challenge.

## Retry the exchange

After satisfaction, retry the same STS exchange and add `challenge_id` and `challenge_response`. The response secret is the challenge response value:

```bash
curl -X POST "$CARACAL_STS_URL/oauth/2/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data-urlencode "grant_type=urn:ietf:params:oauth:grant-type:token-exchange" \
  --data-urlencode "zone_id=$CARACAL_ZONE_ID" \
  --data-urlencode "application_id=$CARACAL_APPLICATION_ID" \
  --data-urlencode "client_secret=$CARACAL_APP_CLIENT_SECRET" \
  --data-urlencode "subject_token=$SUBJECT_TOKEN" \
  --data-urlencode "subject_token_type=urn:ietf:params:oauth:token-type:access_token" \
  --data-urlencode "resource=resource://payments" \
  --data-urlencode "challenge_id=$CHALLENGE_ID" \
  --data-urlencode "challenge_response=$CHALLENGE_SECRET"
```

The STS consumes the challenge atomically. A second retry with the same `challenge_response`, a retry for a different resource set, an expired challenge, or a retry before satisfaction returns 401.

## SDK behavior

The OAuth packages surface `interaction_required` as a typed exception, but the current TypeScript, Python, and Go clients expose only the challenge ID and do not accept `challenge_response` in exchange options. For flows that must complete challenge consumption, perform both the initial challenge-producing exchange and the retry directly against the STS token endpoint so the caller can retain `challenge_secret`.

### TypeScript

```typescript
import { OAuthClient, InteractionRequiredError } from '@caracalai/oauth';

const oauth = new OAuthClient(stsUrl, zoneId, applicationId);

try {
  await oauth.exchange(subjectToken, 'resource://payments', { clientSecret });
} catch (err) {
  if (err instanceof InteractionRequiredError) {
    console.log(`Satisfy challenge ${err.challengeId}, then retry through the STS token endpoint.`);
  }
}
```

`InteractionRequiredError` properties:

| Property | Type | Description |
| -------- | ---- | ----------- |
| `code` | `'interaction_required'` | Always this value |
| `challengeId` | `string` | Challenge ID to pass to the satisfaction API |
| `resource` | `string \| undefined` | Resource the challenge is for |
| `acrValues` | `string \| undefined` | Authentication context requirement, when present |

### Python

```python
from caracalai_oauth import InteractionRequiredError

try:
    await oauth.exchange(subject_token, 'resource://payments', opts)
except InteractionRequiredError as err:
    print(f'Satisfy challenge {err.challenge_id}, then retry through the STS token endpoint.')
```

## Polling pattern

Between receiving the challenge and retrying the exchange, the caller needs to know when the challenge is satisfied. Two patterns work:

### Webhook or event notification (preferred)

Your satisfaction portal notifies the caller via a callback URL when the user completes the MFA flow. The caller resumes the exchange immediately.

### Polling loop

If a callback is not feasible, poll the STS challenge status endpoint:

**TypeScript:**
```typescript
async function waitForChallengeSatisfaction(
  challengeId: string,
  stsUrl: string,
  maxWaitMs = 300_000,
  pollIntervalMs = 2_000,
): Promise<void> {
  const deadline = Date.now() + maxWaitMs;
  while (Date.now() < deadline) {
    const res = await fetch(`${stsUrl}/step-up/${challengeId}`);
    if (res.status === 404) throw new Error(`Step-up challenge ${challengeId} was not found`);
    const status = await res.json() as { satisfied?: boolean };
    if (status.satisfied) return;
    await new Promise((resolve) => setTimeout(resolve, pollIntervalMs));
  }
  throw new Error(`Step-up challenge ${challengeId} not satisfied within ${maxWaitMs}ms`);
}
```

**Python:**
```python
import asyncio
import time

async def wait_for_challenge(
    challenge_id: str,
    sts_url: str,
    http_client,
    max_wait_s: int = 300,
    poll_interval_s: float = 2.0,
) -> None:
    deadline = time.monotonic() + max_wait_s
    while time.monotonic() < deadline:
        response = await http_client.get(f'{sts_url}/step-up/{challenge_id}')
        if response.status_code == 404:
            raise RuntimeError(f'Step-up challenge {challenge_id} was not found')
        status = response.json()
        if status.get('satisfied'):
            return
        await asyncio.sleep(poll_interval_s)
    raise TimeoutError(f'Challenge {challenge_id} not satisfied within {max_wait_s}s')
```

## Challenge rate limiting

The STS enforces a rate limit on failed challenge attempts per principal:

- After 5 failed attempts within a 2-minute window, the principal enters a 5-minute cooldown.
- During cooldown, all exchange attempts for that principal on that resource return HTTP 429 with event type `challenge_cooldown`.
- After the cooldown expires, the attempt counter resets.

A failed attempt is an exchange where the challenge secret is wrong or the challenge was already consumed.

## Challenge types

| Type | Who satisfies it | Typical trigger |
| ---- | ---------------- | --------------- |
| `mfa` | End user completing MFA in portal | Sensitive resource access |
| `human_approval` | Approver in a workflow system | High-value or irreversible action |
| `software_attestation` | Automated attestation service | Workload security assertion |

The challenge type is determined by the policy's `step_up_required` diagnostic value.

## Audit events

| Event type | Meaning |
| ---------- | ------- |
| `token_exchange` with `diagnostics: [{step_up_required: "mfa"}]` | Policy triggered step-up, challenge created |
| `challenge_invalid` | Exchange retry failed — wrong secret, expired, or already consumed |
| `challenge_cooldown` | Principal is in rate-limit cooldown |
| `token_exchange` with `challenge_resolved: true` | Successful retry after challenge satisfaction |
