Resources are the protected targets in a zone. Each resource has a string identifier, a list of scopes, an optional upstream URL for Gateway routing, and an optional credential provider for downstream authentication. Providers describe the credential source the Gateway may use when an authenticated, user-bound provider grant exists.

The default product path is Gateway-mediated HTTP: register the resource with an upstream URL, attach a provider when the upstream needs OAuth or API-key authentication, and let the Gateway enforce Caracal access before it brokers the provider call. Other flows should be described as connector-verified services, application-managed external provider calls, or future native broker plugins.

Conceptually, a resource plus its provider configuration forms a Gateway route. Operators should reason about routes and connector-verified services with five explicit choices:

| Connector field | Current source of truth |
| --- | --- |
| Protocol | The upstream URL and integration surface, such as REST, MCP-over-HTTP, gRPC-over-HTTP, webhook, or a connector-verified service |
| Auth mode | `caracal_jwt` when no provider is attached; brokered OAuth/OIDC/API-key when a supported provider grant is attached |
| Gateway mode | Gateway-mediated when `upstream_url` and `gateway_application_id` are set; connector-verified otherwise |
| Identity forwarding | Off by default; enabled only by provider config `forward_caracal_identity: true` for trusted sidecars or controlled MCP servers |
| Audit mode | Gateway-mediated routes record authorization and execution outcome; connector-verified services must emit their own result audit if they need action-proof records |

## Provider flow matrix

Use this matrix to set expectations before registering a provider or integration:

| Protocol/provider style | Status | Credential handling | Recommended path |
| --- | --- | --- | --- |
| HTTP REST / JSON API | Implemented default | Caracal token to Gateway; optional brokered OAuth/OIDC/API key upstream | Register a resource with `upstream_url` and call through Gateway |
| MCP over HTTP you control | Implemented default | Caracal token to Gateway; optional `X-Caracal-Identity` only when explicitly enabled | Register as a Gateway resource |
| Express, FastMCP, Go `net/http` services you own | Supported pattern | Service verifies Caracal token and consumes revocation stream | Use connector middleware only when Gateway-mediated routing is not the right boundary |
| OAuth2/OIDC provider credentials | Implemented Gateway behavior | Provider token remains inside Gateway/broker path | Attach provider and user-bound provider grant |
| API-key provider credentials | Implemented Gateway behavior | API key remains inside Gateway/broker path and is substituted into the configured header | Attach `apikey` provider with `header_name` or `auth_header` |
| Third-party provider SDKs that hide transport | Application-managed attribution unless wrapped | Provider SDK must not receive Caracal internals or expose provider secrets to agents | Route traffic through Gateway if possible; otherwise attach Caracal context and document the enforcement gap |
| gRPC, queues, webhooks, external MCP servers you cannot modify | Adapter-required supported pattern | Adapter enforces Caracal verification, revocation, trace, and result audit | Build an enforcement adapter before production use |
| Workload identity federation | Future direction in OSS | Registration metadata only | Do not rely on automatic Gateway credential minting yet |

Do not document or operate an integration as Gateway-supported unless it has end-to-end registration, credential brokering, revocation, and action-result audit coverage.

## Scope vocabulary

Scopes are strings for OAuth and JWT compatibility, but operators should design them as a stable typed vocabulary:

| Pattern | Use for |
| --- | --- |
| `<domain>:<action>` | Product permissions such as `calendar:read`, `payments:submit`, `inventory:export` |
| `<domain>:<object>:<action>` | Narrower resource families such as `control:zone:write` |
| Provider-native scopes | Only when the upstream provider already defines the vocabulary and policy maps it explicitly |

Do not place user IDs, request IDs, secrets, or one-off parameters in scopes. Use policy input, delegated-permission restrictions, grants, or provider configuration for dynamic bounds.

## Prerequisites

- `CARACAL_ADMIN_TOKEN` set.
- A zone ID available (`CARACAL_ZONE_ID` or `--zone <id>`).

## Register a resource

```bash
Open the terminal management interface and follow the matching view.
  --identifier "resource://inventory" \
  --scopes "inventory:read,inventory:write,inventory:admin" \
  --name "Inventory Service"
```

The `--identifier` value is the tool/provider resource name agents request access to. It appears in issued token claims and policy input.

To include an upstream URL for Gateway routing:

```bash
Open the terminal management interface and follow the matching view.
  --identifier "resource://inventory" \
  --scopes "inventory:read,inventory:write" \
  --name "Inventory Service" \
  --upstream-url "http://inventory:8080" \
  --gateway-application-id "app-gateway"
```

The Gateway uses the upstream URL to forward requests after verifying the Caracal access token. `gateway_application_id` is required with `upstream_url`; it names the same-zone application identity the Gateway uses for its internal STS exchange. Creating or patching this resource automatically creates or updates the Gateway provider route. Without an upstream URL, the resource can still be used for token issuance but the Gateway has no destination to route to.

### Resource options

| Flag | Description |
| ---- | ----------- |
| `--identifier` | Required. The tool/provider resource string (e.g., `resource://payments`) |
| `--scopes` | Required. Comma-separated list of valid scope strings |
| `--name` | Human-readable display name |
| `--upstream-url` | URL the Gateway forwards matching requests to |
| `--gateway-application-id` | Same-zone application identity used by Gateway for this resource |
| `--prefix` | If set, the Gateway uses prefix matching rather than exact path matching |
| `--provider` | Attach an existing credential provider ID |

## Register a credential provider

Providers define credential metadata for upstream authentication. Registration supports `oauth2`, `oidc`, `apikey`, and `workload`, but Gateway brokered forwarding currently has explicit runtime behavior only for:

| Kind | Runtime behavior |
| --- | --- |
| `oauth2` / `oidc` | Forwards a user-bound delegated provider access token as a bearer-style provider credential. Refresh is supported when the active provider grant has an encrypted refresh token and provider config includes `token_endpoint`. |
| `apikey` | Forwards an encrypted provider grant value as an API-key credential using the required `header_name` or `auth_header` from provider config. Caracal does not mint API keys; the key must already exist in the provider-grant storage path. |
| `workload` | Registration metadata only in the current OSS runtime. Workload assertion minting is not brokered by STS/Gateway yet. |

If no active provider grant exists for the configured resource/provider pair, STS denies the Gateway exchange instead of falling back to a different provider or exposing provider credentials to the caller. Malformed provider config is rejected; API-key providers must declare the exact upstream header before STS returns a brokered directive.

By default, provider-native forwarding does **not** send the Caracal access token to the upstream. Add `"forward_caracal_identity": true` only for upstreams you control, such as an internal sidecar or controlled MCP server that intentionally verifies `X-Caracal-Identity`. Leave it unset for third-party SaaS providers.

When the Gateway brokers a provider credential, action audit records the resource, upstream host, provider auth mode, and non-secret provider/grant identifiers. It does not log provider tokens, request bodies, or response bodies by default.

### API key provider

```bash
Open the terminal management interface and follow the matching view.
  --identifier "provider://inventory-apikey" \
  --name "Inventory API Key" \
  --kind apikey \
  --config @provider-config.json
```

`provider-config.json` for an API key provider:

```json
{
  "api_key": "<provider-api-key>",
  "header_name": "X-Api-Key",
  "forward_caracal_identity": false
}
```

The `api_key` field matches the credential encryption regex (`/(secret|password|token|api[_-]?key|…)/i`), so it is encrypted at rest under the zone KEK.

`header_name` and `auth_header` are mutually equivalent for API-key forwarding. One of them must be present and must be a valid HTTP header name.

### OAuth2 provider

```bash
Open the terminal management interface and follow the matching view.
  --identifier "provider://payments-oauth" \
  --name "Payments OAuth2" \
  --kind oauth2 \
  --config @oauth2-config.json
```

`oauth2-config.json`:

```json
{
  "token_endpoint": "https://auth.payments.example.com/oauth/token",
  "allowed_token_hosts": ["auth.payments.example.com"],
  "client_id": "svc-caracal",
  "client_secret": "super-secret-value",
  "scopes": ["payments.read", "payments.write"],
  "forward_caracal_identity": false
}
```

### OIDC provider

```bash
Open the terminal management interface and follow the matching view.
  --identifier "provider://sso-oidc" \
  --kind oidc \
  --config @oidc-config.json
```

`oidc-config.json`:

```json
{
  "issuer": "https://sso.example.com",
  "client_id": "caracal-gateway",
  "client_secret": "oidc-secret",
  "audience": "https://api.example.com"
}
```

### Workload provider

The workload provider kind records metadata for mTLS or workload identity federation (e.g., AWS IAM roles for service accounts). The current OSS STS/Gateway path does not mint workload assertions from this registration; use it for inventory and future connector integration, not for automatic Gateway credential substitution.

```bash
Open the terminal management interface and follow the matching view.
  --identifier "provider://aws-workload" \
  --kind workload \
  --config @workload-config.json
```

## Attach a provider to a resource

After creating the provider, attach it to the resource:

```bash
Open the terminal management interface and follow the matching view.
  --provider prov-def456
```

Or set it at resource creation time with `--provider prov-def456`.

## Complete example: resource with OAuth2 provider

```bash
# 1. Create the OAuth2 provider
Open the terminal management interface and follow the matching view.
  --identifier "provider://payments-oauth" \
  --name "Payments OAuth2" \
  --kind oauth2 \
  --config @oauth2-config.json
# → ID: prov-abc123

# 2. Create the resource
Open the terminal management interface and follow the matching view.
  --identifier "resource://payments" \
  --scopes "payment:read,payment:submit,payment:void" \
  --name "Payments API" \
  --upstream-url "http://payments:8080" \
  --gateway-application-id "app-gateway" \
  --provider prov-abc123
# → ID: res-def456

# 3. Verify
Open the terminal management interface and follow the matching view.
```
