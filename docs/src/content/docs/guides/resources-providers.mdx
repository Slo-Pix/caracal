---
title: Define Resources and Providers
description: Register protected upstream resources and optional provider credentials from the Console.
---

Resources tell Caracal what can be protected. Providers describe the upstream credential shape a Gateway-routed resource may use, including the internal Caracal mandate flow.

## When to create each object

| Object | Create it when |
| --- | --- |
| Resource | You need a stable protected target with scopes and a Gateway route to an upstream. |
| Provider | The target needs an explicit upstream auth mode: Caracal mandate, OAuth2 authorization code, OAuth2 client credentials, API key, or bearer-token upstream auth. |
| Gateway application | You want the Gateway to front an HTTP upstream and issue the final mandate for that request. |

## Console workflow

1. Start the local runtime with `caracal up`.
2. Open `caracal console`.
3. Select `resource`.
4. Create a resource with a stable identifier, such as `https://api.example.com/payments`.
5. Add action-oriented scopes, such as `payments:read` and `payments:refund`.
6. Set the upstream URL and Gateway application for the protected service.
7. Select `Caracal mandate` for internal services that verify Caracal tokens, or choose a provider-native type when the upstream needs external credentials.
8. Attach the provider to the resource if applicable.

## Provider auth modes

| Provider type | Required main fields | Runtime behavior |
| --- | --- | --- |
| Caracal mandate | No credential fields. | Gateway forwards the Caracal resource mandate in `Authorization: Bearer ...`; the internal service verifies issuer, audience, scopes, target, expiry, and revocation. |
| OAuth2 authorization code | Authorization endpoint, token endpoint, redirect URI, client ID, client secret. | Uses user/resource provider grants and refreshes delegated provider tokens inside STS. |
| OAuth2 client credentials | Token endpoint, client ID, client secret. | STS obtains a service-to-service provider token for Gateway upstream calls. |
| API key | Header name and API key. | Gateway forwards the sealed provider key in the configured header. |
| Bearer token | Bearer token. | Gateway forwards the sealed provider token through `Authorization` unless another header is configured. |

Provider secrets are sealed at creation time and are not returned by list or detail APIs. Caracal mandate providers have no provider-native secret; they make the internal mandate-forwarding choice explicit while keeping the same Gateway and verifier checks.

## Identifier and scope guidance

| Field | Good pattern |
| --- | --- |
| Resource identifier | URL-shaped or URN-shaped stable target. |
| Scope | `domain:action`, such as `tickets:read`. |
| Upstream URL | The internal service URL the Gateway can reach. |
| Gateway application | The application identity Gateway uses for upstream exchanges. |
| Provider identifier | Stable name for the upstream credential system. |

Do not use mutable deployment hostnames as policy identifiers unless they are the actual authority boundary. Keep the identifier stable even if the upstream URL changes.

## Validate the setup

- Open the resource in the Console and confirm scopes are present.
- Create a grant for the application and subject that will request the scopes.
- Activate a policy set that allows the resource.
- Send a Gateway request and inspect the audit event. For Caracal mandate upstreams, also confirm the internal service uses a Caracal verifier or connector.
