---
title: Run an Agent with caracal run
description: Use a generated runtime profile to inject scoped Caracal access tokens into a subprocess.
pageType: workflow
---

`caracal run` starts your command with resource tokens that were exchanged from a confidential Caracal application. The simplest path is to let the Console generate the runtime profile after first setup, then store the one-time client secret in the secret file named by that profile.

## Generate the profile

Open the Console and choose **guided setup**:

```bash
caracal console
```

Keep **runtime profile** enabled. The result view shows:

- the profile path
- the secret-file path
- the generated TOML profile
- the token environment variable derived from your resource ID
- the one-time application client secret, masked by default

Enable **write files** when you want the Console to create the generated profile and one-time secret file locally. Leave it disabled when platform automation or a secret manager will write those files. If you copy the output manually, write the profile to the shown path and write the client secret to the shown secret-file path with owner-only permissions.

## Run the workload

Set `CARACAL_CONFIG` to the generated profile path, then run the command you normally use for the workload after `caracal run --`.

At runtime, Caracal reads the profile, exchanges each configured resource credential through STS, and injects each access token into the environment variable shown in the Console result. Your process reads that variable and sends the token to the protected service or Gateway.

## Gateway-routed resources

If first setup included an upstream URL, send requests through Gateway with:

- `Authorization: Bearer` followed by the injected resource token
- `X-Caracal-Resource` set to the resource ID from the Console result

Gateway verifies the token, checks revocation, evaluates the active policy set, proxies to the configured upstream, and writes audit evidence for the request.

## Finish with audit explanation

After the first successful protected call, open **audit** in the Console. Select the request and use **explain** with its request ID. The explanation should show the application, resource, scopes, active policy version, policy set version, and final decision. If the call fails, use the same explanation path to identify whether the failure came from token exchange, policy, Gateway routing, or upstream response.
