Policies decide whether the STS may issue a mandate. They are written in Rego, versioned immutably, bundled into policy sets, and activated per zone.

## Policy objects

| Object | Purpose |
| --- | --- |
| Policy | Named Rego document with immutable versions. |
| Policy version | A specific content hash and schema version. |
| Policy set | A named bundle of policy versions. |
| Policy set version | A specific manifest of policy versions. |
| Active policy set version | The version the STS evaluates for a zone. |

## Decision contract

Caracal policies return a `result` object from `package caracal.authz`.

```text
package caracal.authz

import rego.v1

default result := {
  "decision": "deny",
  "evaluation_status": "complete",
  "determining_policies": [],
  "diagnostics": [],
}

result := {
  "decision": "allow",
  "evaluation_status": "complete",
  "determining_policies": [{"policy": "baseline-scopes"}],
  "diagnostics": [],
} if {
  # input.principal.id identifies the acting user, service, or agent.
  # input.resource.identifier names the protected resource.
  every scope in input.context.requested_scopes {
    scope in data.allowed_scopes[input.principal.id][input.resource.identifier]
  }
}
```

Example input values:

| Input | Example |
| --- | --- |
| `input.principal.id` | `user:finance-operator@example.com` |
| `input.application.id` | `app_lynx_control` |
| `input.resource.identifier` | `lynx/mercury-bank` |
| `input.context.requested_scopes` | `["payments:read", "payments:write"]` |
| `input.delegation_edge` | Present when authority was delegated from another agent session. |

## Common input areas

| Input area | Contains | Common check |
| --- | --- | --- |
| `input.principal` | Acting subject or agent identity. | Match a user, service, or agent allowlist. |
| `input.application` | Registered application requesting exchange. | Restrict a client to approved resources. |
| `input.resource` | Target resource and requested scopes. | Compare requested scopes with resource policy. |
| `input.context` | Session, requested scopes, exchange metadata, and challenge state. | Require step-up or session freshness. |
| `input.grant` | Matching grant data when available. | Enforce grant expiry and scope bounds. |
| `input.delegation_edge` | Delegation path and typed constraints when exchange is delegated. | Enforce hop count, chain match, and constraint limits. |

## Policy outcomes

| Outcome | Effect |
| --- | --- |
| `allow` | STS signs a mandate if token and session checks also pass. |
| `deny` | STS refuses the exchange and records diagnostics. |
| step-up diagnostic | STS returns `interaction_required` with a challenge ID. |

Step-up is expressed as a diagnostic such as `{"step_up_required": "mfa"}`. The STS creates a challenge and expects a later exchange with the challenge proof.

## Authoring rules

- Keep policies deterministic and side-effect free.
- Use the Console or Admin API validation endpoint before activation.
- Activate through a policy set version, not by editing active content in place.
- Use diagnostics to explain denies and step-up triggers.
- Keep reusable decision logic in separate policy versions only when it improves ownership or review.
