## Definition

The **Ledger** is Caracal's authoritative record of every authority decision. It is append-only: events are written but never modified or deleted within the retention window.

The ledger is populated via the `caracal.audit.events` Redis stream. The STS, gateway, and coordinator all publish events. The audit service (`services/audit`) consumes the stream and persists events to PostgreSQL.

Events are scoped to a [Workspace](/concepts/workspace/). Querying the ledger for a workspace returns the full history of authority decisions made within that workspace.

## Why it exists

Authority enforcement without a record is not enforceable. The ledger provides:

- **Accountability** — every action that ran through the gateway is traceable to a principal, a mandate, and a declared intent.
- **Forensics** — after an incident, the full sequence of authority decisions is available for analysis.
- **Compliance** — the ledger satisfies audit requirements that demand a record of who authorized what, when.
- **Replay** — a historical session can be re-run against a current or past policy to verify that the same decisions would be made today.

## Event types

| Event | Published by | Description |
|---|---|---|
| `mandate.issued` | STS | A mandate was issued. Includes principal ID, intent, caveats, TTL. |
| `mandate.denied` | STS | A mandate request was denied by policy. Includes principal ID, intent, deny reason. |
| `mandate.verified` | Gateway | A mandate was verified on a gateway request. Includes JTI, request path, upstream. |
| `mandate.expired` | Gateway | A mandate was rejected because it had expired. |
| `mandate.revoked` | STS | A session revocation was issued. All mandates for this session are invalidated. |
| `replay.detected` | Gateway | A JTI was seen more than once. The request was rejected. |
| `delegation.issued` | Coordinator | A delegation was granted to a sub-agent. |
| `delegation.revoked` | Coordinator | A delegation was revoked. |
| `policy.step_up` | STS | A step-up authentication challenge was issued. |

## Lifecycle

Ledger entries flow through two stages:

1. **Stream** — events are published to the `caracal.audit.events` Redis stream. The stream acts as a buffer and provides at-least-once delivery to the audit service.
2. **Persist** — the audit service consumes the stream and writes to PostgreSQL. Events remain in the stream for the configured Redis stream retention window (default: 24 hours) to support lag recovery.

## Querying the ledger

```bash title="Query recent ledger events for a workspace"
# List the last 100 authority events for a workspace
caracal audit list --workspace my-workspace --limit 100

# Explain a specific mandate by its JTI
caracal audit explain --jti jti_01HZXYZ...

# Replay a session against the current policy
caracal audit replay --session-id sess_01HZABC...
```

## Constraints and limits

- The ledger is append-only within the retention window. Events cannot be modified or deleted.
- Retention window is configurable per workspace. Default: 90 days for PostgreSQL persistence. The Redis stream buffer is shorter (default: 24 hours).
- Replay is read-only: replaying a session does not issue new mandates or make real API calls.
- Cross-workspace ledger queries are not supported. Each workspace's ledger is isolated.
