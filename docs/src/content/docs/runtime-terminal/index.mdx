Caracal ships two terminal-facing tools:

- **`caracal`** — thin runtime shell for stack lifecycle, readiness checks, purging local state, token-injected process execution, and launching the management surface.
- **`caracal-terminal`** — the terminal management interface for human operational workflows across zones, applications, resources, providers, policies, grants, sessions, agents, audit events, diagnostics, and Control API lifecycle.

The runtime shell intentionally does not include product-management commands. Use `caracal terminal` or `caracal-terminal` for interactive management, and use the Control API for automation that must manage Caracal remotely.

Both tools are compiled with Bun for Linux (x64, arm64), macOS (x64, arm64), and Windows (x64). The runtime is self-contained; no separate Node.js installation is required.

---

## Authentication and endpoints

The terminal management interface communicates with the control-plane API and discovers operator credentials from the same places as the engine:

| Variable | Default | Purpose |
| --- | --- | --- |
| `CARACAL_ADMIN_TOKEN` | Auto-discovered when possible | Bearer token for control-plane management routes. |
| `CARACAL_API_URL` | `http://localhost:3000` in development | Control-plane API base URL. |
| `CARACAL_ZONE_ID` | From `caracal.toml` or saved terminal state | Default zone for zone-scoped views. |
| `CARACAL_COORDINATOR_URL` | `http://localhost:4000` in development | Coordinator service URL for agent and delegation views. |
| `CARACAL_COORDINATOR_TOKEN` | — | JWT with `agent:lifecycle` authority for agent and delegation views. |
| `CARACAL_CONFIG` | Auto-resolved | Explicit `caracal.toml` path. |

`CARACAL_ADMIN_TOKEN` is discovered in order: `CARACAL_ADMIN_TOKEN`, `CARACAL_ADMIN_TOKEN_FILE`, the runtime secret file written by `caracal up`, and `infra/secrets/files/caracalAdminToken` in development mode.

## Supported surfaces

| Area | Surface | Purpose |
| --- | --- | --- |
| Stack lifecycle | `caracal up`, `caracal down`, `caracal status`, `caracal purge` | Start, stop, probe, and reset the local stack. |
| Workload execution | `caracal run -- <command>` | Inject scoped resource tokens into subprocesses. |
| Human management | `caracal terminal` or `caracal-terminal` | Manage Caracal entities interactively. |
| Remote automation | Control API | Drive approved management actions through authenticated HTTP. |

`caracal run` is a runtime execution command, not a management workflow. The terminal management interface does not run workloads; use the top-level runtime shell for token-injected processes.

---

## Exit behavior

Runtime shell commands write errors to stderr and exit `0` on success or `1` for validation, readiness, or API failures. `caracal run` returns the subprocess exit code after credentials are acquired.

The terminal management interface requires a controlling TTY. In non-interactive environments it exits `1` and directs automation to the Control API or SDK.
