## Can Caracal replace secrets-in-env for agents?

Yes. That is the core use case.

Instead of handing an agent a long-lived API key or `.env` secret, the agent
asks Caracal for short-lived, scoped authority at the moment it acts. The
request is checked against policy, the credential is brokered at the Gateway so
the agent never holds it, and both the decision and the result are recorded for
audit and revocation.

| Property | Secret in environment | Caracal |
| --- | --- | --- |
| Lifetime | Long-lived, often unrotated | Short-lived, per request or session |
| Scope | Whatever the key allows | Narrowed by policy to the action |
| Agent holds the credential | Yes | No, the Gateway attaches it |
| Revocation | Rotate the key everywhere | Revoke the session centrally |
| Audit | None by default | Decision and result recorded |

If you do not have agents or automated workflows touching credentials, you
probably do not need Caracal yet.

## When Caracal is not the right fit

- You only have human users behind a normal login. Use an IdP, not Caracal.
- You have no agents or automation touching credentials or provider APIs.
- You need a general secrets vault for static config. Caracal brokers
  authority for actions; it is not a replacement for a config store.

See [What Caracal Is Not](../what-caracal-does/#what-caracal-is-not) for the
full boundary.

## Prefer a managed option

Everything here is the self-hosted open-source edition. If you would rather not run the Gateway and services yourself, or you need managed multi-tenancy, a hosted management UI, and SSO, see the [Enterprise Edition](/enterprise/).
