Use this path when you installed the released `caracal` and `caracal-console` binaries. It keeps the first run focused on product workflows instead of repository tooling.

## Prerequisites

- Docker 24+ with the Compose v2 plugin
- `caracal` and `caracal-console` installed from [Installation](../installation/)

## Start Caracal

```bash
caracal up
caracal status --ready
caracal console
```

`status --ready` must pass before setup. If it fails, run `caracal status --ready --json` and check the local stack logs.

## Create the first protected path

In the Console, choose **guided setup**. Enter the real zone name, agent application name, protected resource ID, scopes, and upstream URL you want to protect. Leave optional fields blank when the resource does not need Gateway routing or a provider credential yet.

The result view gives you the real zone ID, application ID, resource ID, client secret, policy set, and generated runtime profile. Store the one-time client secret in the secret-file path shown by Console, copy the generated profile to the profile path shown by Console, and keep the file permissions owner-only.

Set `CARACAL_CONFIG` to the profile path shown by Console, then run the workload command through `caracal run --`. Do not use copied placeholder profiles for onboarding; generate a profile from the objects you just created.

Stop the stack with:

```bash
caracal down
```

Use [Five-Minute Setup](../five-minute-setup/) for the full protected HTTP request and audit explanation flow. Use [Contributor Quickstart](../contributor-quickstart/) only when you are working from the source tree.
