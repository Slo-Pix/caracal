This quickstart brings up the full Caracal OSS stack on your local machine using Docker, verifies all services are healthy, and creates a local zone ready for integration.

:::note[Contributors]
This page covers the **installed binary** workflow. If you are building Caracal from source, follow [Local Setup](../contributing/setup/) instead.
:::

## What you will run

The stack consists of seven services:

| Service | Port | Purpose |
| ------- | ---- | ------- |
| PostgreSQL | 5432 | Persistent state (zones, policies, sessions, audit events) |
| Redis | 6379 | Event streams, revocation signals, JTI replay tracking |
| STS | 8080 | Token exchange with OPA policy evaluation |
| API | 3000 | Control-plane REST — zones, policies, resources, grants |
| Gateway | 8081 | MCP reverse proxy with mandate verification |
| Audit | 9090 | Audit event consumer and ledger writer |
| Coordinator | 4000 | Agent session lifecycle and delegation graph |

All ports are bound to `127.0.0.1` only. They are not exposed on external interfaces.

## Step 1: Install the CLI

Install `caracal` and `caracal-tui` via the [Installation](./installation/) page, then verify:

```sh
caracal --version
```

## Step 2: Start the stack

```sh
caracal up
```

On first run, `caracal up` automatically:

1. Creates a runtime home directory for Caracal:
   - **Linux/WSL**: `~/.local/share/caracal/` (or `$XDG_DATA_HOME/caracal/`)
   - **macOS**: `~/Library/Application Support/caracal/`
2. Writes a `.env` file there with all required secrets randomly generated — database passwords, the admin token, encryption keys. You do not need to set any of these manually.
3. Pulls versioned container images and starts all seven services.

The startup sequence is:

1. PostgreSQL and Redis start and pass health checks.
2. Database migrations run once and complete.
3. STS, API, Gateway, Coordinator, and Audit start in dependency order.

First startup takes 30–60 seconds while Docker pulls images and PostgreSQL initializes.

:::caution
Do not manually edit the runtime `.env` file unless you are rotating secrets. All values are generated by `caracal up` and are consistent with the running containers. Editing them out of sync will break authentication between services.
:::

## Step 3: Check service health

```sh
caracal status
```

You should see all five application services reporting healthy:

```
api            3000  ✓ ok      200
sts            8080  ✓ ok      200
gateway        8081  ✓ ok      200
audit          9090  ✓ ok      200
coordinator    4000  ✓ ok      200
```

If a service is not healthy within two minutes, check its logs:

```sh
docker logs caracal-sts-1 --tail 50
```

## Step 4: Provision a zone and application

The stack starts with an empty database. Create a zone, register an application, and capture the generated client secret. The admin token is read from the runtime `.env` automatically — no manual export is needed.

```sh
caracal zone create --name local
# → captures the new zone id, e.g. zone_abc123

caracal app create --zone zone_abc123 --name local-app
# → prints the application id and the client secret (shown only on creation)
```

Register a resource the application will exchange tokens for, and activate a policy that permits the exchange. See [Define Resources and Providers](../../guides/resources-providers/) and [Author a Rego Policy](../../guides/author-policy/) for the full schema.

Author `caracal.toml` in your project directory with the values from the previous step:

```toml
zone_url = "http://localhost:8080"
zone_id  = "zone_abc123"

application_id     = "app_def456"
app_client_secret  = "<client secret from caracal app create>"

[[credentials]]
env      = "RESOURCE_TOKEN"
resource = "resource://my-api"

[mcp_governance]
mode = "block"
```

Set the file mode to `0600` — it contains your application's client secret.

```sh
chmod 600 caracal.toml
```

## Step 5: Verify the token exchange

With the stack and `caracal.toml` in place, request a token for the registered resource:

```sh
caracal credential read resource://my-api
```

If successful, a JWT is printed to stdout. This is a mandate — an ES256-signed token issued by the STS after the active policy permitted the exchange.

Decode it to inspect the claims:

```sh
caracal credential read resource://my-api | cut -d. -f2 | base64 -d | jq .
```

You should see claims including `zone_id`, `scope`, `target`, and an expiry (`exp`) 15 minutes from now.

## Stopping the stack

```sh
caracal down
```

This stops and removes containers but preserves the named volumes (`postgresData`, `redisData`). State persists across restarts.

## What is running

At this point you have:

- A fully operational seven-service Caracal stack
- A zone with a registered application, resource, and active policy
- A `caracal.toml` that the CLI and SDKs can read for configuration

Continue to [First Integration](./first-integration/) to wire an application to Caracal using the SDK.
