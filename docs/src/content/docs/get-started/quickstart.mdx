This quickstart brings up the full Caracal OSS stack, verifies service health, and prepares the default Gateway-first flow: a zone, an agent app, a tool/provider resource, and an active policy set. You operate it as one local stack even though the internals are split into separate trust-boundary services.

:::note[Contributors]
This page covers the **installed binary** workflow. If you are building Caracal from source, follow [Local Setup](../contributing/setup/) instead.
:::

## What you will operate

For local adoption, Caracal has one operator surface:

```sh
caracal up
caracal status
caracal down
```

Those commands manage the containers, secrets, health checks, and startup order. You do not need to start services manually for the default local path.

## What runs inside the stack

The stack consists of five Caracal services plus Postgres and Redis:

| Service | Port | Purpose |
| ------- | ---- | ------- |
| PostgreSQL | 5432 | Persistent state (zones, policies, sessions, audit events) |
| Redis | 6379 | Event streams, revocation signals, JTI replay tracking |
| STS | 8080 | Public token exchange with OPA policy evaluation |
| API | 3000 | Control-plane REST — zones, policies, resources, grants |
| Gateway | 8081 | Default tool/provider execution plane |
| Audit | 9090 | Audit event consumer and ledger writer |
| Coordinator | 4000 | Agent session lifecycle and delegation graph |

All ports are bound to `127.0.0.1` only. They are not exposed on external interfaces. The service split exists to preserve security boundaries; the local operating model is still one stack.

## Step 1: Install the CLI

Install `caracal` and `caracal-tui` via the [Installation](./installation/) page, then verify:

```sh
caracal --version
```

## Step 2: Start the stack

```sh
caracal up
```

On first run, `caracal up` automatically:

1. Creates a runtime home directory for Caracal:
   - **Linux/WSL**: `~/.local/share/caracal/` (or `$XDG_DATA_HOME/caracal/`)
   - **macOS**: `~/Library/Application Support/caracal/`
2. Generates secret files under `$CARACAL_HOME/secrets/` (database passwords, admin token, encryption keys). Services consume them via the `*_FILE` convention; no secret strings are written to any `.env` file.
3. Writes `$CARACAL_HOME/caracal.env`, an operator override template with every tunable entry commented out. Uncomment a line to override the built-in default (issuer URL, S3 export, resource limits, etc.). Mode/version/registry are pinned and rejected if set.
4. Pulls versioned container images and starts the five Caracal services plus Postgres and Redis.

The startup sequence is:

1. PostgreSQL and Redis start and pass health checks.
2. Database migrations run once and complete.
3. STS, API, Gateway, Coordinator, and Audit start in dependency order.

First startup takes 30–60 seconds while Docker pulls images and PostgreSQL initializes.

:::caution
Do not edit files under `$CARACAL_HOME/secrets/` unless you are rotating secrets — they back the running stack. Operator overrides go in `$CARACAL_HOME/caracal.env`, which is safe to edit.
:::

## Step 3: Check service health

```sh
caracal status
```

You should see all five application services reporting healthy:

```
service      port   mode    status  detail
-------      ----   ----    ------  ------
api          3000   health  ✓ ok    200
sts          8080   health  ✓ ok    200
gateway      8081   health  ✓ ok    200
audit        9090   health  ✓ ok    200
coordinator  4000   health  ✓ ok    200
```

Then run dependency-aware readiness:

```sh
caracal status --ready
```

Readiness checks include Postgres, Redis, STS reachability, Gateway bindings, audit replay, and stream consumers. If a service is not ready, the `detail` column includes the failing reason, for example `503 postgres_unreachable` or `503 stream_consumers_not_ready`. Keep that reason with any support or incident notes before checking logs.

## Step 4: Protect one HTTP resource

The stack starts with an empty database. Create the product objects needed for a Gateway-mediated call: zone, Gateway app, resource route, policy, policy set, activation, grant, and local config values. The admin token is read from `$CARACAL_HOME/secrets/caracalAdminToken` automatically — no manual export is needed.

The copy-paste flow below uses `jq` to capture IDs from JSON output and `openssl` to generate the local app secret.

```sh
APP_CLIENT_SECRET="cs_$(openssl rand -hex 32)"
ZONE_ID="$(caracal zone create --name "Local Development" --slug local-development | jq -r '.id')"
APP_ID="$(caracal app create \
  --zone "$ZONE_ID" \
  --name local-gateway-app \
  --credential-type token \
  --client-secret "$APP_CLIENT_SECRET" \
  --traits gateway | jq -r '.id')"
RESOURCE_ID="$(caracal resource create \
  --zone "$ZONE_ID" \
  --identifier "resource://local-tool" \
  --name "Local Tool" \
  --upstream-url "https://httpbin.org" \
  --gateway-application-id "$APP_ID" \
  --scopes "demo:read" | jq -r '.id')"

cat > gateway-scope.rego <<'REGO'
package caracal.authz

import rego.v1

default allow := false
default result := {"decision": "deny", "evaluation_status": "complete", "determining_policies": [], "diagnostics": []}

result := {"decision": "allow", "evaluation_status": "complete", "determining_policies": [{"policy": "gateway-scope"}], "diagnostics": []} if {
  allow
}

allow if {
  every scope in input.context.requested_scopes {
    scope in input.resource.scopes
  }
}
REGO

POLICY_VERSION_ID="$(caracal policy create \
  --zone "$ZONE_ID" \
  --name "Gateway Scope Policy" \
  --description "Allows requested scopes registered on the target resource." \
  --file gateway-scope.rego \
  --schema-version 2026-05-20 | jq -r '.version.id')"
POLICY_SET_ID="$(caracal policy-set create \
  --zone "$ZONE_ID" \
  --name "Gateway Policies" \
  --description "Local Gateway policy set" | jq -r '.id')"
POLICY_SET_VERSION_ID="$(caracal policy-set version "$POLICY_SET_ID" \
  --zone "$ZONE_ID" \
  --policy-versions "$POLICY_VERSION_ID" | jq -r '.id')"
caracal policy-set activate "$POLICY_SET_ID" --zone "$ZONE_ID" --version "$POLICY_SET_VERSION_ID" >/dev/null
caracal grant create --zone "$ZONE_ID" --app "$APP_ID" --user local-user --resource "$RESOURCE_ID" --scopes "demo:read" >/dev/null
rm gateway-scope.rego
```

Write `caracal.toml` in the working directory. The `app_client_secret` is an application secret; store it as you would any other service credential.

```sh
cat > caracal.toml <<EOF
zone_url = "http://localhost:8080"
zone_id = "$ZONE_ID"

application_id = "$APP_ID"
app_client_secret = "$APP_CLIENT_SECRET"

[[credentials]]
env = "RESOURCE_TOKEN"
resource = "resource://local-tool"

[mcp_governance]
mode = "block"
EOF
chmod 600 caracal.toml
```

The file should look like this:

```toml
zone_url = "http://localhost:8080"
zone_id = "zone_abc123"

application_id = "app_def456"
app_client_secret = "cs_generated_once"

[[credentials]]
env = "RESOURCE_TOKEN"
resource = "resource://local-tool"

[mcp_governance]
mode = "block"
```

The Gateway application identity is same-zone service identity used for internal STS exchange. In production, use a dedicated Gateway application identity and review policy content before activation.

Run the readiness doctor if the credential step below fails:

```sh
caracal doctor --extended
```

It checks API reachability, admin authentication, selected zone, resource registration, active policy sets, grants, audit queryability, service readiness, and operator metrics. Use `--json` when you want to attach the output to an incident record or CI gate.

## Step 5: Verify scoped access

With the stack and `caracal.toml` in place, request a short-lived access token for the registered resource:

```sh
caracal credential read resource://local-tool
```

If successful, a signed Caracal access token is printed to stdout. This proves the public STS allowed the agent app to access the resource; provider-native credentials are not returned to the CLI.

Inspect it to review the claims:

```sh
caracal credential read resource://local-tool | caracal credential inspect -
```

You should see `verification=not_verified` because this command decodes locally instead of validating the signature. The claims should include `zone_id`, `scopes`, `resource`, and an expiry 15 minutes from now.

To test the full product path, call the resource through the Gateway:

```sh
caracal run -- sh -c 'curl -i -sS \
  -H "Authorization: Bearer $RESOURCE_TOKEN" \
  -H "X-Caracal-Resource: resource://local-tool" \
  "http://localhost:8081/get"'
```

You should receive a `200` response from `https://httpbin.org/get`. The Gateway verifies the Caracal token, resolves the provider route, forwards the upstream call, and records action-result audit.

## Stopping the stack

```sh
caracal down
```

This stops and removes containers but preserves the named volumes (`postgresData`, `redisData`). State persists across restarts.

## What is running

At this point you have:

- A fully operational local Caracal authority plane
- A zone with an agent app, tool/provider resource, active policy set, grant, and Gateway provider route
- A `caracal.toml` that the CLI and SDKs can read for configuration

Continue to [First Integration](./first-integration/) to wire an application to Caracal using the SDK.
