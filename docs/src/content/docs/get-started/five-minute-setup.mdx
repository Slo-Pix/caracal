This path starts the released local stack, creates one protected resource from the Console, calls it through Gateway, and explains the audit evidence. Use a real upstream service that is reachable from the Gateway container or process.

## Prerequisites

- Docker 24+ with the Compose v2 plugin
- `caracal` and `caracal-console` installed from [Installation](../installation/)

## Start the stack

```bash
caracal up
caracal status --ready
```

`status --ready` must return success before you configure resources. If it fails, inspect `caracal status --ready --json` and the service logs from Docker Compose.

## Configure the first resource with Console

Open the Console:

```bash
caracal console
```

In the Console:

1. Choose **guided setup**.
2. Enter the zone name and agent application name used by the team that will own the workload.
3. Enter the resource ID and scopes for the upstream you are protecting.
4. Enter the real upstream URL when you want Gateway routing.
5. Enter a real first request path when you want Console to generate an exact Gateway check.
6. Turn on **write files** when you want Console to create the runtime profile and secret file on this machine.
7. Keep **advanced options** closed for the first run unless you need provider credentials, custom profile paths, a custom token environment variable, skipped policy activation, or file overwrite behavior.

The result view shows the created objects, the one-time client secret, the active policy set, the generated runtime profile, local setup commands, file-write status, and first-success commands. If file writing was skipped, run the local setup commands, paste the revealed one-time client secret as the only line in the secret file, and keep the profile and secret files owner-readable only.

## Call the protected resource

Use the generated `CARACAL_CONFIG=... caracal run --` command prefix with the real command that starts your workload.

The token environment variable in the generated profile is derived from the resource ID you entered. Use that injected token when your workload calls Gateway or the protected service. For Gateway-routed resources, the result view includes an exact `curl` command when first request path was provided; otherwise it shows the required Gateway URL and headers so you can run the request against a real protected path.

## Explain the first success

Return to the Console **audit** view after the first request. Select the request and open **explain** with its request ID. The explanation should show the created application, protected resource, scopes, policy version, policy set version, Gateway result, and final decision. If the request failed, use the same explanation to identify whether the next correction belongs in runtime config, policy, resource routing, or the upstream service.

## Shut down

```bash
caracal down
```
