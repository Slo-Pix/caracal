This path starts a local authority plane, creates one protected resource from the Console, calls it through Gateway, and explains the audit evidence. Use a real upstream service that is reachable from the Gateway container or process.

## Prerequisites

- Docker with Compose v2
- Node 24, pnpm 11.1.1, Bun 1.3.14
- Go 1.26 and Python 3.14 if you plan to run the full test suite

Install dependencies:

```bash
corepack enable
corepack prepare pnpm@11.1.1 --activate
pnpm install --frozen-lockfile --prefer-offline
```

## Start the stack

```bash
pnpm caracal up
pnpm caracal status --ready
```

`status --ready` must return success before you configure resources. If it fails, inspect `pnpm caracal status --ready --json` and the service logs from Docker Compose.

## Configure the first resource with Console

Open the Console:

```bash
pnpm caracal-console
```

In the Console:

1. Choose **first-setup**.
2. Enter the zone name and agent application name used by the team that will own the workload.
3. Enter the resource ID and scopes for the upstream you are protecting.
4. Enter the real upstream URL when you want Gateway routing.
5. Leave provider ID blank unless the resource already has a provider credential source.
6. Keep **activate policy** and **runtime profile** enabled for the first run.

The result view shows the created objects, the one-time client secret, the active policy set, and a generated runtime profile. Store the client secret in the secret-file path shown by Console and copy the profile content to the shown profile path.

## Call the protected resource

Set `CARACAL_CONFIG` to the profile path shown by Console, then run your workload command through `pnpm caracal run --`.

The token environment variable in the generated profile is derived from the resource ID you entered. Use that injected token when your workload calls Gateway or the protected service. For Gateway-routed resources, include `Authorization: Bearer` with the injected token and `X-Caracal-Resource` with the resource ID shown by Console.

## Explain the first success

Return to the Console **audit** view after the first request. Select the request and open **explain** with its request ID. The explanation should show the created application, protected resource, scopes, policy version, policy set version, Gateway result, and final decision. If the request failed, use the same explanation to identify whether the next correction belongs in runtime config, policy, resource routing, or the upstream service.

## Shut down

```bash
pnpm caracal down
```
