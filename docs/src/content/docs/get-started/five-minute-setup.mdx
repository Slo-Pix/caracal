title: Five-Minute Setup
description: Protect one HTTP resource, call it through the Gateway, and explain the audit trail.

---

This walkthrough creates one Caracal-protected resource and verifies the first call. Use a real upstream URL reachable from the local Gateway container or process.

## 1. Start the stack

```bash
caracal up
caracal status --ready
caracal console
```

Create product objects only after readiness succeeds.

## 2. Run guided setup

In the Console, choose **guided setup**.

| Wizard step | Recommended first-run choice |
| --- | --- |
| Zone | Create a local evaluation zone or select an existing one. |
| Agent app | Create a confidential app for the workload. |
| Provider | Create a provider only if the resource needs a brokered upstream credential. |
| Resource | Enter a stable resource identifier, required scopes, and Gateway upstream details when the resource fronts an external API. |
| Access policy | Let the Console create and activate the starter policy. |
| Review | Enable **write profile files** if this machine should receive the runtime profile and secret file. |

The result view shows the zone ID, application ID, resource ID, app client secret, active policy set, profile path, secret-file path, generated profile, and first-success commands.

If the Console does not write files, create the files manually from the result view and keep them owner-readable only. The app client secret is shown once for a newly created app.

## 3. Run the workload

Use the generated profile path with the command that starts your workload:

```bash
CARACAL_CONFIG=/path/to/caracal.toml caracal run -- ./start-agent.sh
```

`caracal run` exchanges the configured application secret for resource-specific Caracal access tokens and injects them into the environment variables named in the profile. The workload reads those variables and presents the token to the Gateway or protected service.

For SDK-based workloads, the same generated profile is read by `Caracal.connect()` or `caracal.Connect()`.

## 4. Call through the Gateway

For a Gateway-routed HTTP resource, send the request to the Gateway URL from the result view and include:

| Header | Value |
| --- | --- |
| `Authorization` | `Bearer <Caracal access token>` |
| `X-Caracal-Resource` | The resource ID from guided setup. |

If the wizard generated an exact `curl` command, run that first. Otherwise use the real protected path you configured.

## 5. Explain the result

Return to the Console:

1. Open **audit**.
2. Find the request you just made.
3. Press Enter on the event or open **explain** and paste its request ID.

The explanation should show the application, resource, requested scopes, policy version, policy set version, Gateway result, and final decision. If the request failed, the explanation tells you whether to fix runtime config, policy, resource routing, or the upstream service.

## Optional repeatable validation

Use the repository validator when you already have a real short-lived mandate and need a CI-friendly proof that STS, Gateway, routing, and metrics work together:

```bash
CARACAL_TOKEN="$(cat ./runtime-token.txt)" \
node scripts/validateFirstSuccess.mjs \
  --gateway-url http://localhost:8081 \
  --sts-url http://localhost:8080 \
  --resource "$RESOURCE_ID" \
  --token-env CARACAL_TOKEN \
  --path /real/protected/path
```

The script checks STS and Gateway readiness, sends a real Gateway request, and confirms the Gateway success counter advanced. It does not create objects or mint tokens.

## Shut down

```bash
caracal down
```

Next, use [First Integration](./first-integration/) to wire the same profile into application code.
